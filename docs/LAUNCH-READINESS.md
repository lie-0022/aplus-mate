# A+ Mate 파일럿 런칭 준비 (Launch Readiness)

> 목표: **한 수업의 실제 학생(20~50명)을 받아 파일럿 운영**. 교수 1명 + 그 수업 학생들.
> 소프트 런칭(소규모 코호트)이 원칙 — 공개 전 첫 사용자들과 밀착 피드백.

---

## 런칭 체크리스트 (5 버킷)

리서치 프레임(MVP 런칭 = 제품/기술/법무/온보딩/운영)을 우리 맥락에 맞춤. 판정은 코드 점검(Explore) 결과로 채운다.

### 1. 제품 — 핵심 플로우 end-to-end
파일럿의 핵심 플로우 하나가 막힘없이 돌아야 한다:
```
가입(Google) → 프로필 작성 → 수업 코드로 참여 → 모집/커넥트로 매칭 → 팀 활동 → (팀플이면) 동료평가
```
- [ ] 이 플로우가 신규 계정으로 처음부터 끝까지 되는지 (실제 화면 검증)
- [ ] "2분 내 aha" — 학생이 들어와서 곧 자기 팀/할 일을 만나는지

### 2. 기술·안정성
- [ ] **에러 모니터링** — 프로덕션 에러를 실제로 볼 수단(로그/알림). 최근 tRPC 예약어로 502 부팅 실패를 겪음 → 부팅 크래시 예방·감지
- [ ] **롤백 경로** — 문제 배포 시 되돌리기 (백업 브랜치·revert)
- [ ] 클라 ErrorBoundary / 서버 에러 핸들링 / 사용자에게 보이는 에러 메시지
- [ ] 부팅 스모크 테스트(로컬 prod 부팅)로 배포 전 크래시 차단

### 3. 법무·컴플라이언스 (한국 PIPA) ⭐런칭 블로커
- [ ] **개인정보처리방침**(Privacy) 실 내용 — 수집 항목·목적·보유기간·제3자/위탁·파기·권리
- [ ] **이용약관**(Terms) 실 내용
- [ ] **회원가입/프로필 작성 시 동의 절차** — 필수/선택 구분, 동의 거부권 고지
  - 수집 항목(우리): 필수 = 이름·학교·학과·학년, 구글 계정(이메일·식별자) / 선택 = 카카오 오픈채팅 URL·스킬 태그
  - 목적 = 같은 수업 팀 매칭·팀 활동
  - 보유기간 = 회원 탈퇴 시 파기(`users.deletedAt` 존재)
  - 카카오 오픈채팅: **팀원에게 공개**됨 → 수집·이용 동의 시 명시(제3자 제공 아님, 서비스 내 공개)
- [ ] 만 14세 미만 별도(대학생 대상이라 실질 무관, 문구만)

### 4. 온보딩 — 빈 상태에서 길잡이
- [ ] 처음 들어온 학생이 뭘 할지 아는지 (Dashboard 온보딩 체크리스트, 빈 상태 CTA)
- [ ] 프로필 미완성/수업 미참여 상태에서 막다른 길 없는지
- [ ] 교수용 "학생 초대 방법"(조인 코드) 안내가 명확한지

### 5. 운영·데이터
- [ ] **프로덕션 데모/테스트 데이터 정리** — `google:demo-*` 유저, `DEMO-*` 수업, 데모대학교, 테스트 계정(친구들). 실학생과 섞이면 혼란 → 런칭 전 정리 수단(`clearDemoData`) 실행
- [ ] **첫 유저 세팅** — 첫 교수에게 professor 권한 부여(`admin.setUserRole`), 그 교수가 수업 등록·조인코드 발급
- [ ] **신고/문의 창구** — 신고(reports) + 운영자 연락처 노출 위치
- [ ] 소프트 런칭 규모(20~50명) 안내

## 갭 & 우선순위 (점검·구현 반영)

| 순위 | 갭 | 상태 |
|---|---|---|
| P0 | 평가 데이터 처리 동의 | ✅ 구현 — TeamEvaluate 익명 배너 + 제출 시 `consents.record("evaluation")` 기록 |
| P0 | 약관/개인정보 확정 | ✅ **PIPA 문구 보강**(수집항목·목적·기간·제3자·권리·철회·책임자·개정) / ⚠️ 확대 전 **법률 검수는 사용자** |
| P1 | 첫 커넥트 UX | ✅ Dashboard 온보딩에서 수업 있으면 그 수업으로 직행 |
| P1 | ErrorBoundary 영어 | ✅ 한국어화 + 스택 DEV 한정 + 홈/문의 버튼 |
| P1 | 학교 이메일 정책 | ⚠️ **사용자 결정** — 무제한(A, 조인코드로 통제) vs 도메인 화이트리스트(B) |
| P2 | 신고 UI | ✅ 이미 완성(ReportDialog: 게시글·댓글) |
| P2 | 지원·법적 링크 노출 | ✅ AppLayout 아바타 메뉴에 문의·처리방침·약관 |
| P2 | 프로덕션 데모 정리 | ⚠️ **런칭 직전 승인 후** `clearDemo` 실행 |
| P2 | 네트워크 재시도 | ⏳ 후속(비차단) |

**남은 것 = 사용자 결정 3개**: 약관 최종 검수(확대 시), 이메일 정책(A/B), 데모 정리 승인. 코드 갭은 모두 메움.

## 런칭 D-day 절차 (초안)
1. 데모 데이터 정리(clearDemo) → 프로덕션 깨끗이
2. 개인정보·약관 게시 + 동의 절차 배포
3. 첫 교수 계정 professor 지정 → 수업 등록 → 조인 코드 발급
4. 신규 계정으로 end-to-end 스모크 테스트(화면)
5. 교수가 학생에게 코드 공유 → 학생 유입 → 밀착 모니터링
