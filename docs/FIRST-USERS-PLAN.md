> 목표: 5년 로드맵이 아니라, **이미 배포된 MVP 위에서 백석대 한 분반에 실제 학생을 받아 핵심 루프를 한 번이라도 돌리고, 그것이 "운영자가 손으로 민 데모"가 아니라 "제품이 만든 수요"였는지를 (가능한 범위에서) 측정**하는 것.

## 1. 첫 실사용자의 정의 — 측정 + 그 한계의 정직한 명시

### 1-1. 산출물 마일스톤 (단계 분리)
- **1차(D 세션, ~1주차 말):** 한 분반에서 같은 course 행 등록·프로필완성 **8~12명**, 그중 **최소 2쌍(4명)**이 매칭 수락→팀 생성→카카오 오픈채팅 실연결.
- **2차 후행(실제 팀플 마감 +2~4주):** 성공 기준 = **"teams completed + 양쪽 evaluations 제출 완료(evaluationStatus=done)"**. 배지 발급은 점수 의존이라 통제 불가.

### 1-2. 검증 — 교란 통제 시도 + 식별력 한계 명시
> ⚠ **솔직한 전제(라운드3 #3):** n=8~12 단일 코호트에서는 어떤 측정도구도 통계적 식별력을 거의 못 준다. 대조군을 두면 이미 작은 표본을 분모 3~6으로 쪼개 이항검정 검정력이 사실상 0이고, 콘시어지 대상이 운영자 친분풀로 자기선택돼 처치-교란과 친분-교란이 엉키며 n이 작아 층화도 불가하다(기록≠통제). **따라서 1차 코호트는 "통계적 증명"이 아니라 "치명적 실패의 조기 발견 + 2차 재현을 위한 가설 다듬기"로 목적을 한정한다.** 이 한계는 §9에 잔존성으로 등재한다.

그 한계 안에서 할 수 있는 통제:
- **n=1 자발 행동은 "성공"이 아니라 "신호 후보"**로만 기록. 단일 코호트 절대값으로 성공 선언 금지.
- **교란량 기록(통제 아님, 사후 해석용):** 참석자 중 운영자 친분 비율, 인센티브 수령 여부, "incentive_forced_match"(§D 책임매칭 페어) 표시.
- **준식별자 위험:** name 마스킹해도 학과+학년+대학+스킬태그로 재식별 가능 → 동의문(§A1)에 명시.

### 1-3. 피벗 트리거 (검정력 한계를 인정한 형태)
- 실패선을 절대 임계가 아니라 **사전 등록한 방향성**으로: "동원 인원 대비 자발행동이 베이스라인 기대 이하 + 2차 분반에서도 재현 실패"일 때 피벗. **단일 코호트 0만으로 피벗하지 않는다**(표본 한계 때문).
- **선결 조건 1 — 재참여 채널 생존(§1-4):** 채널이 죽어 있었으면 "수요 없음"이 아니라 "재방문 수단 없음"이므로 피벗 보류.
- **선결 조건 2 — 에타 교란 배제(§A1):** 자생 0의 원인이 "에타가 이미 더 싸게 해결"이면 *포지셔닝* 문제로 처리.
- **2차 분반 재현을 "성공 선언의 필수 조건"으로 승격**(라운드3 #3): 1차에서 신호가 보여도 2차에서 재현돼야 "수요"로 인정.

### 1-4. 재참여 채널 — 핑과 자생 측정의 분리 (라운드3 newHigh #3)
- 카카오 오픈채팅방을 재참여 채널로 재활용: 운영자가 **D+7에 후속 핑** 1회.
- **핑은 그 자체로 개입이므로, 핑 수신자는 사전에 "핑 처치군"으로 분리**한다. **자생 측정 윈도우는 D+0~D+7(핑 이전)으로 한정**하고, 핑 이후 행동은 자생 카운트에서 제외(핑 반응률은 별도 지표). → "자발"과 "핑잔효"가 섞이지 않게.

## 2. 핵심 전략 — 한 문장 (포지셔닝 교정)

> **"N인 조 편성"이 아니라 "같은 강의실에서 아직 모르는 한 명과 1:1로 먼저 연결"**(코드는 dyad만 생성). 운영자가 현장에서 양면 동의로 성사시키고 앱이 그 합의를 기록한다.

- 카피·세션 스크립트에서 "팀원 찾기/조 편성"(3~5인 함의)을 **"1:1 짝 연결"**로 재포지셔닝. "학생이 1:1 dyad를 실제 원하는가"는 A1에서 직접 묻고, 최종은 §9 잔존비약.
- **첫 코호트 신뢰 원천 = "운영자 개인 평판"**(배지는 0). self-serve 확산 시 대체재는 §9.
- 코드가 막아주는 것(매칭 후에만 연락처 공개)은 살리고, 뚫린 것(무음 함정·솔로 백지·딥링크 등록 갭·PII)은 학생 올리기 전에 메운다.
- 미구현 기능(보고서/헤이영/.ac.kr/스마트매칭)은 **광고하지 않는다**.

        │
Phase C 운영 준비 (개발 0~소)           C1 표준course+딥링크  C2 가시성  C3 안전망·PII·이의제기·SLA(+?ref 수기)
        │
Phase D 현장 세션 (B+C 완료 직후, 부스)   → 첫 매칭·팀·연락
        │

- **B 일정 재산정(라운드3 cheap#7):** v2의 "당일~1.5일"은 과소추정. B3 consents 신규테이블+db:push만 1~1.5일, 여기에 B1 백필·B2 enrollment 리팩터·C3 PII·신규 테스트가 더해지면 **현실적으로 2~3일**. consents CREATE TABLE은 백필/마스킹과 분리된 단계로 런북에 두고 staging db:push diff 선검토. **이 일정이 D-2 하드게이트를 위협하면 B3(법적)·B1·B2(퍼널 파손)를 먼저, C3 PII·attribution은 D 세션 후로 미룬다.**

## 4. 단계별 실행계획

### Phase A — 개발 착수 전 결정/검증 (0원, 1~2일)

#### A1. 수요·채널·동원·대안 동시 검증
| 체크 | 행동 | 합격 기준 |
|---|---|---|
| 채널 실재 | 접근 가능한 단일 분반/단톡방 **이름까지 특정** | 분반 1개 + 글 올릴 권한 |
| 물리 공간 | 강의실 세션이면 **점유·교수 묵인·시간충돌 부재** 확인 | 교수/조교 1인 사전 동의 |
| 페인 + **에타 대안** | 5명 인터뷰: "팀 못 정했나?" + **"지금 어디서 구하나(에타/단톡)? 에타에 같은 분반 팀구함 글 있나? 부족한 점은 정확히 뭔가?"** | ≥2 "팀 못 정함" + **에타 대비 구체적 결핍 명문화** |
| dyad 수요 | "한 명이 더 필요한가, 셋이 더 필요한가?" | dyad 수요 ≥2 (아니면 §9 등재) |
| 타이밍 | 조 편성이 **D 세션 주차에** 진행 | 마감 D-2 이후 |
| 동기 | "운영자가 직접 짝 매칭 베타" 참여 의향 | ≥5명 |
| 인센티브 비용 | 커피·"책임매칭"의 **비용 주체·상한** 확정 | 1코호트 상한 명시(예산 0원과 충돌하므로 실예산 항목으로) |
| **2차 분반 확보** (라운드4) | 성공 선언이 2차 재현에 의존하므로(§1-3), **동일 학기 내 또다른 비교가능 분반**(조 편성이 다른 D 세션 주차에 진행)에 접근·동원 가능한지 | 1차 특정 시점에 2차 후보 ≥1 식별 (불가 시 §9 #9로 등재) |

- 타깃: 친분 풀 얕은 분반(전공기초 대형/편입·전과 다수/캡스톤) 우선. 친한 친구끼리 짜는 강의 회피.
- **A1 'No'면:** 분반/페인/타이밍 교체. 에타 결핍 미확인 시 *포지셔닝* 재검토.

| 변수 | 검증 |
|---|---|

### Phase B — 최소 코드 게이트 (2~3일, 우선순위 순)

#### B1. 프로필 함정 정합화 (0순위)
- `db.ts:115`에서 **`kakaoOpenChatUrl` 제거** + **`skillTags` 조건도 제거**(ProfileSetup이 skillTags 미수집이라 default `[]` truthy에 우연 의존, 라운드3 cheap#2) → **`university && department && year`만**.
- **백필 SQL을 런타임 게이트와 동일 조건으로:** `UPDATE users SET profileCompleted=true WHERE university IS NOT NULL AND department IS NOT NULL AND year IS NOT NULL`. (런타임/백필/컬럼 세 곳 정합 — 불일치 시 레거시 행이 다음 수정에서 무음 미완성 회귀.)
- **kakao는 게이트에서 빼되, 연락수단 0 함정은 비대칭 인지하에 처리(라운드3 cheap#4):** 매칭 흐름이 비대칭이라 "수락 직후 모달"은 수신자만 잡는다. → **이상적으론 `TeamDetail` 첫 진입 시 자기 kakao 미입력이면 모달**(양쪽 포착). 단 **요청자가 TeamDetail을 자발 방문한다는 보장이 없다**(accept에 navigate 없음·알림 0). → **권장: 커넥트 전송 시점(요청자)에 kakao를 받고, 수신자는 수락 화면에서 받는다.** TeamDetail 모달은 Dialog+useState+profile.update+invalidate 신규 배선이라 "1시간"이 아닌 **반나절** 작업. 1차 현장 코호트는 운영자가 그 자리서 양쪽 kakao를 채워 가리고, **비동기 요청자 kakao 도달은 §9 잔존**으로.
- **딥링크 직진입 게이트 갭(라운드3 cheap#3):** App.tsx:53 리다이렉트는 `location==='/'`에서만, ProtectedPage는 profileCompleted 미검사 → `/courses/:id` 직진입 미완성 유저는 ProfileSetup 우회→createMatchRequest throw. **B1에 "딥링크 직진입은 게이트 안 함"을 명시**하고, self-serve 차단 필요시 ProtectedPage에 profileCompleted 가드(returnTo 보존) 추가를 B 항목으로.

#### B2. enrollment-aware CourseDetail — 단일 리팩터 (0순위, 라운드3 #1·#2 통합)
> v2가 B2(솔로백지)·B5(enroll 버튼)·?ref를 독립 나열했으나 **실제론 하나의 작업**이다. 미등록 딥링크 방문자는 `courses.students` throw(routers.ts:140) → CourseDetail isError 분기 없음(273-283) → **빈 탭**(throw로 isLoading=false, 스켈레톤·빈상태 둘 다 안 뜸). enroll 버튼만으론 안 닫힌다.

하나의 enrollment-aware 리팩터로:
1. **`courses.students` 쿼리를 `courses.myCourses`로 등록 확인 후에만 `enabled`**(클라가 등록여부를 아는 유일한 수단 — `isEnrolled` 전용 쿼리 없음) + **isError 분기 추가**(throw 대비).
2. **미등록 시 enroll CTA를 우선 렌더**(Courses.tsx의 `enrollMutation` 62·288 재사용).
3. **본인 제외 후 길이로 빈상태 판정**: `others = students.data?.filter(s=>s.user.id!==user?.id) ?? []` → `others.length===0`일 때 초대 카드. **isError(미등록) 케이스도 별도 분기**(여기엔 enroll CTA).
4. **enroll onSuccess에서 `courses.students`·`courses.get`·`courses.myCourses` invalidate** → 등록 직후 같은 화면에서 팀탭 활성화·커넥트까지 끊김 없이. (등록 여부는 `courses.myCourses`로 판별 — `isEnrolled` 전용 쿼리는 라우터에 없음.)

#### B2-DL. cold 딥링크 경로 보존 + 수락 루프 종결 (0순위, 라운드4 신규)
> v3 이전은 딥링크 게이트를 "로그인된 미완성 유저"만 봤다. **더 근본 결함**: 로그아웃 학생이 `/courses/:id` 딥링크 클릭 → 로그인 → OAuth 콜백이 무조건 `/`로(oauth.ts:47) → 딥링크 경로 통째 소실. **C1/D2가 배포하는 바로 그 first-touch가 cold 신규 유저에게 코드상 도달 불가**, A2 self-serve 측정도 성립 불가.

- **cold 딥링크 보존 — 정확한 위치(라운드5 정정):**
  - **저장 위치:** "getLoginUrl 직전"이 아니다. 로그아웃 유저가 `/courses/:id`를 치면 **`ProtectedPage`(App.tsx:22-30)가 `!user`를 보고 *먼저* `setLocation("/")`로 튕긴 뒤** Home에서 `getLoginUrl()`이 호출되므로, getLoginUrl 직전엔 이미 경로가 `/`다. → **`ProtectedPage`의 `!user` 분기에서 `setLocation("/")` *직전에* `localStorage.setItem("returnTo", pathname+search)`** 해야 `/courses/123`이 잡힌다.
  - **복원 위치:** OAuth 콜백은 모두 `/`로 착지(oauth.ts:47). `/`의 분기는 **`App.tsx:52-60`(Router landing redirect)**: 프로필 완성→/dashboard, 미완성→/profile/setup. → **프로필 완성 복귀 유저(가장 흔한 경우)의 returnTo는 `App.tsx:52-60`에서 소비**해야 한다(ProfileSetup엔 안 옴). 미완성 유저는 `App.tsx`가 returnTo를 보존한 채 `/profile/setup`으로 보내고 **`ProfileSetup.tsx:34`의 하드코딩 `setLocation("/dashboard")`도 returnTo로 교체**. 두 지점의 순서: App.tsx가 returnTo를 *clobber하지 않고* 넘기도록.
  - (localStorage가 외부 OAuth 왕복에서 살아남는지는 문제없음 — useAuth.ts:45·ThemeContext·DashboardLayout이 이미 동일 origin 콜백에서 localStorage를 왕복함. 문제는 캡처 *시점·위치*였다.)
- **수신자 navigate(1줄):** `acceptMutation.onSuccess`에서 반환 `teamId`로 `setLocation(\`/teams/${teamId}\`)`(데이터 이미 존재, 무비용) → 수락 즉시 팀 화면.
- **요청자 수락-신호(구조적 부재):** 요청자에게 수락을 알릴 프리미티브가 0이라 **1차 동기 코호트는 운영자가 현장에서 요청자에게 수락을 구두 확인**(런북). 그 전까지 §5의 "자발 커넥트" 지표는 운영자 개입 없이 닫히지 않으므로 **측정 무효 단서로 §5·§9에 명시**. 확산 단계엔 최소 수락 알림 + "보낸 요청" 뷰 신설을 후속 B로(§9 #7).
- **미구현 시:** §9에 "cold 딥링크 self-serve 미지원, 1차는 운영자 현장 네비게이트로 대체"를 명문화하고 A2 self-serve 지표를 "현장 동반 완주"로 한정.

### Phase C — 운영 준비 (개발 0~소규모)

#### C1. 표준 course 선생성 + 딥링크
- 운영자가 표준 표기로 course 1건 선생성. 학생에겐 `/courses/{id}` 딥링크 배포(**B2 리팩터로 그 화면에서 등록·커넥트가 실제 되도록 봉합한 뒤에만 유효**).

#### C2. 운영자 가시성
- 1차(동기성 세션)는 notifyOwner 불필요. **비동기 보강 기본 = prod DB `SELECT * FROM team_matches WHERE status='pending'` 직접 조회(런북)**, 실명-카톡 매핑은 운영자 시트.
- **notifyOwner 오귀속 교정(라운드3 cheap#5):** notifyOwner는 routers에서 호출 0 + owner/admin 전용 + forge 미설정시 throw라 **학생을 찌르는 수단이 아니다.** "폴백 시 켠다"가 아니라 **"운영자가 카톡으로 학생을 수동으로 찌른다"**로 서술.

#### C3. 안전망·PII·이의제기·attribution·SLA
- **PII:** `getCourseStudents`·`getReceivedMatchRequests`·`getCoursePosts.author.name`(db.ts:277) payload에서 name 제거/이니셜. **단 "실명 제거 ≠ 식별 차단"** — 준식별자(학과+학년+대학+스킬) 재식별 위험은 동의문에 명시, 필요시 커넥트 전 카드에서 학과/학년 단과대 수준 범주화 검토.
- **평가 이의제기:** TeamEvaluate에 "보복성 저평가는 신고 대상" 고지 + 운영자 이메일/폼(2인 단일 평가자 분쟁 대비).
- **신고/차단·안전수칙:** 약관·푸터·TeamDetail.
- **?ref attribution 격하(라운드3 #1):** OAuth 전체 리다이렉트가 ?ref 쿼리를 파괴하고 schema/라우터/localStorage에 캡처 경로 0이라 **서버 자동 귀속은 미구현**. → **1차는 "?ref = 운영자 수기 대조용(수집편향 존재)"으로 §5에 명시 격하**, "시스템 자동 포착" 인상 제거. **실구현(딥링크 랜딩 localStorage 저장→OAuth 왕복 복원 + userCourses에 invitedByUserId 컬럼 + courses.enroll/enrollCourse에 ref 입력)은 "self-serve 확산 단계" 항목으로.**
- **운영자 부재 SLA:** pending/신고 응답 목표(예 48h) 사전 고지 + **백업 운영자 1명**에 prod DB 조회·신고 위임. 부재 기간 신규 세션 금지.
- **마이그레이션 런북:** B1 백필·B3 consents를 staging 리허설→영향 행 SELECT→트랜잭션+롤백→2인 검토.

### Phase D — 현장 세션 → 첫 매칭 (B+C 완료 직후, 부스 방식)

| 단계 | 행동 | 게이트 |

> ⚠ **자발 커넥트 지표 무효 단서(라운드4):** 요청자에게 수락을 알릴 코드 프리미티브가 없어(§0 매칭 행), "자발 커넥트→수락→연결" 루프는 **운영자 현장 구두 확인 없이는 닫히지 않는다.** 즉 "운영자 비동원 자생 신호"를 1차에서 깨끗이 측정할 수 없다. 이 한계가 §9 #7로 등재돼 있으며, 확산 단계의 수락 알림+"보낸 요청" 뷰가 생기기 전까지 이 지표는 **참고치**로만 본다.

## 9. 잔존 비약 — 실측 의존 (측정 도구는 장착, 단 식별력 한계 명시)

1. **표본 검정력 한계:** n=8~12 단일 코호트에선 대조군·비율임계·층화의 통계적 식별력이 사실상 0. 1차는 "증명"이 아니라 "치명적 실패 조기 발견 + 2차 가설 다듬기". **2차 분반 재현이 성공 선언의 필수 조건**(§1-3).
2. **dyad 수요 진위:** 학생이 "1:1 짝"을 원하는가(아니면 4인 조). A1에서 떠보되 최종은 코호트 행동.
3. **리텐션 존재:** 학기당 저빈도 이벤트의 재방문 동기. §1-4로 측정하되 없으면 단발 도구로 귀결될 수 있음.
4. **신뢰 이전성:** 첫 코호트 신뢰원천=운영자 평판. self-serve 확산에서 대체재(배지 누적/후기/제3 운영자) 미검증.
5. **에타 대비 내재가치:** 보상 끊긴 뒤 에타 network effect를 이길 내재 가치(블라인드 평판 배지)가 첫 코호트엔 0. 채워지는 속도 = 사업 성립 속도.
6. **콜드스타트 확산 비용:** 코호트2+가 운영자 현장 세션 없이 self-serve로 퍼지는지 미검증(매 코호트 O(인력) 우려).
7. **요청자측 수락-신호 루프 종결 프리미티브 부재(라운드4 격상):** `acceptMatch`가 teamId를 반환하나 클라가 버리고(수신자 navigate는 B2-DL에서 1줄 수정), 요청자에게 수락을 알릴 알림/"보낸 요청" 뷰가 0이다. 1차 동기 코호트는 운영자 현장 구두 확인으로 가리되, **이 프리미티브가 없으면 "자발 커넥트→수락" 자생 루프는 측정 자체가 불가**(§5 무효 단서). 확산 단계에 최소 수락 알림 + "보낸 요청" 폴링 뷰 신설 필요. 요청자 kakao 비동기 도달도 이 위에 얹힘.
8. **?ref 서버 귀속(실구현):** 1차는 수기 대조(수집편향). 자동 귀속은 OAuth 왕복 localStorage + 스키마 컬럼(invitedByUserId) + 라우터 입력이 필요한 확산 단계 항목.
9. **2차 분반 동원 가능성(라운드4 신규):** 성공 선언을 2차 분반 재현에 의존(§1-3)시켰으나, 동일 학기 내 비교가능 2차 분반을 운영자가 확보·동원할 수 있는지는 A1에서 확인하되 미검증이면 **재현 게이트가 공허**해진다. self-serve 확산 비용(#6)과 별개의 동원 가정.
