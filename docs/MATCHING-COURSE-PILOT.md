# 매칭 기능 — "한 수업 + 교수 주도" 도입 계획

> 전제(가정): **첫 도입은 한 교수님이 자기 수업 하나를 A+ Mate로 운영**한다. 그 수업 학생들만,
> 교수님을 통해 들어와서, 그 수업 안에서만 팀을 매칭한다. (불특정 다수 자율가입 X)
> 이 전제로 현재 매칭을 재점검하고, 빠진 고리를 메운다.

---

## 1. 시나리오 (end-to-end, 이상적 흐름)

```
[교수]
 1. A+ Mate 가입 → 운영자가 'professor' 권한 부여(또는 교수 인증)
 2. 자기 수업 등록/클레임 → "게임기획기초" (교수 인증 ✓)
 3. 수업의 [조인 코드 / 초대 링크] 발급
 4. 강의 첫 시간/LMS에 "이 수업 팀플은 A+ Mate로 짭니다. 코드 ABC123" 공지
 5. [선택] 팀 구성 마감일 설정: "2026-03-10까지 팀 짜기"

[학생]  (그 수업 학생만)
 6. 가입 → 프로필(학과·학년·스킬) 작성
 7. 조인 코드 입력 or 링크 클릭 → 그 수업에 자동 등록 (검색·생성 불필요)
 8. 같은 수업 학생/모집공고를 보고 → 모집 올리기 or 지원/커넥트
 9. 팀 결성 → 일정·메모·산출물·AI보고서, 끝나면 동료평가

[교수]  (모니터링·촉진)
 10. 대시보드: 참여율 / 팀 구성 현황 / 미배정 학생 / 마감 D-day
 11. 기한 임박 → 미배정 학생에게 독려 알림
 12. 마감 후에도 미배정 학생 → 교수가 개입(자동 매칭 제안 or 수동 배정)
```

이 흐름에서 **6~7번(학생을 그 수업에 깔끔히 모으기)**, **5·10·11·12번(교수 주도권)**이 현재 비어 있다.

---

## 2. 리서치 근거 (외부 베스트 프랙티스)

- **온보딩 = 조인 코드 + 초대 링크가 사실상 표준**: MS Teams(class code), CodeHS(teacher join code), GitHub Classroom(invitation URL), Google Classroom(roster import). → 우리는 **초대 링크만** 있고 **조인 코드 없음**.
- **순수 학생 자율선택은 권장되지 않음(CATME 연구)**: 저학년·서로 모를 때 위험 > 이점. 베스트는 **학생 입력 + 알고리즘/교수 안전망**의 절충. 메트릭에 **수업 시간표(공강)** 가 핵심 → 우리 [[aplus-mate-timetable-plan]]과 연결됨.
- 시사점: 우리 강점(학생이 직접 모집·지원하는 자율 매칭)은 살리되, **교수가 기한·미배정을 챙기는 안전망**을 더한다. 완전 자동배정(CATME식)으로 갈아엎지 않는다.

---

## 3. 현재 구조 — 강점과 갭 (코드 근거)

### 이미 있는 것 ✅
- 교수 수업 소유: `claimCourse`(professorProcedure), `courses.professorId`
- 교수 대시보드: `getCourseDashboard` — 참여율·팀현황·**미배정 학생 명단**·설문 응답률
- 공지·설문 팬아웃 알림, 마일스톤(제출 항목, dueAt)
- 수업 단위 매칭: `teamMatches.courseId` / `recruitments.courseId` 격리
- 초대 링크: `CourseDetail` `handleCopyInvite()` (`/courses/:id`)

### 빠진 것 ❌ (이 전제에서 치명적인 순)
| # | 갭 | 영향 |
|---|---|---|
| G1 | **조인 코드 없음** — 학생이 수업을 직접 검색/생성해야 함 | 교수가 학생을 한 수업에 모으는 게 번거롭고, 같은 수업이 중복 생성될 수 있음 |
| G2 | **교수 인증 배지·등록상태 불명확** | 어느 게 "교수님 정식 수업"인지 학생이 모름 |
| G3 | **매칭 기한 없음** | 교수가 "언제까지 팀 짜라"를 설정·독려 못 함 |
| G4 | **미배정 학생 개입 불가** | 마감 후에도 혼자인 학생을 교수가 못 챙김(보기만 함) |
| G5 | (학생이 수업 생성 가능) | 교수 주도 시나리오와 충돌 — 정책 결정 필요 |

---

## 4. 설계 방향

**원칙**: 학생 자율 매칭(모집공고·커넥트)은 그대로 두고, **교수를 "수업 호스트 + 안전망"으로** 올린다. 한 수업에 깔끔히 모으고(조인코드), 기한으로 리듬을 주고, 미배정은 교수가 마무리.

---

## 5. 단계별 구현 계획 (디테일)

### Phase 1 — 학생 온보딩: 수업 조인 코드 + 교수 인증 (G1·G2) ⭐최우선
**스키마** (`courses` 컬럼 추가, 마이그레이션):
- `inviteCode varchar(8)` UNIQUE — 교수/생성자가 발급. 대문자+숫자 6자(혼동문자 0/O/1/I 제외).

**서버**:
- `courses.generateInviteCode(courseId)` (professorProcedure, 소유 검증) → 코드 생성/회수
- `courses.joinByCode(code, semester)` (protectedProcedure) → 코드로 수업 찾아 `enrollCourse`. 잘못된 코드/이미 등록 처리.
- `getCourseByInviteCode(code)`

**UI**:
- 교수 `Professor.tsx`: 담당 수업에 **"조인 코드: ABC123 · 복사 / 재발급"** 카드. 학생 안내 문구.
- 학생 `Courses.tsx`: 상단에 **"수업 코드로 참여"** 입력칸(코드 → 자동 등록). 검색/생성은 아래로.
- `CourseDetail.tsx`: 헤더에 **교수 인증 배지**(`professorId != null`이면 "교수님 인증 ✓"), 등록 상태 "참여 중 / 참여하기" 명확화.

**검증**: 코드 발급→입력→자동등록 라운드트립. 잘못된/중복 코드 가드.

---

### Phase 2 — 교수 매칭 주도: 팀 구성 마감일 + 독려 (G3) ⭐
**스키마**: `courses.matchingDeadline timestamp NULL`

**서버**:
- `professor.setMatchingDeadline(courseId, deadline|null)` (소유 검증)
- 대시보드에 `matchingDeadline` + 남은 일수 포함
- `professor.nudgeUnassigned(courseId)` — 미배정 학생 전원에게 알림("아직 팀이 없어요! D-3") 팬아웃(공지 알림 패턴 재사용)

**UI**:
- `Professor.tsx` 대시보드: "팀 구성 마감 [날짜 설정]" + "D-day" + **"미배정 N명 독려하기"** 버튼
- 학생 `Dashboard.tsx`/`CourseDetail.tsx`: 마감 D-day 배너(그 수업에 미배정일 때 강조)

---

### Phase 3 — 미배정 학생 마무리: 교수 개입 (G4)
**서버** (둘 중 택1 또는 병행):
- (수동) `professor.placeStudentInTeam(courseId, studentId, teamId)` — 정원·중복 검증, 교수 권한으로 합류
- (반자동) `professor.suggestMatches(courseId)` — 미배정 학생을 스킬/타입으로 묶어 **추천 팀 편성안** 제시(교수가 승인). 완전 자동배정 아님(학생 자율 존중).

**UI**: 대시보드 미배정 명단에서 "팀에 넣기 ▾"(드롭다운으로 기존 팀 선택) 또는 "추천 편성 보기".

> Phase 3는 정책 민감(교수가 강제 배정) → Phase 1·2 검증 후 사용자와 범위 합의하고 진행.

---

### Phase 4 — 정책·UI 정리 (G5)
- 학생의 수업 생성을 **막거나/억제**(교수 주도 모델). 예: 학생은 "코드로 참여"만, 생성은 교수/운영자. (또는 학생 생성 수업은 "미인증"으로 구분)
- 교수 인증 수업 우선 노출, 중복 수업 정리 가이드.

---

## 6. 우선순위 & 이번 범위

| 순위 | Phase | 작업 | 근거 |
|---|---|---|---|
| 1 | **P1** | 조인 코드 + 교수 인증 배지 | 온보딩 표준, 전제의 핵심(학생 모으기) |
| 2 | **P2** | 매칭 마감일 + 독려 | 교수 주도권의 뼈대 |
| 3 | P3 | 미배정 개입 | 안전망 마무리(정책 합의 후) |
| 4 | P4 | 학생 생성 정책 | 모델 정합성 |

**이번에 디벨롭**: P1 → P2 (스키마·서버·UI·검증·배포). P3·P4는 P1·P2 검증 후 사용자와 합의하고 착수.
