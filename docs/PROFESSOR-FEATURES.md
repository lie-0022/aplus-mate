# 교수 페이지 — 설계와 기능

## 운영 모델 (전제)

A+ Mate는 **레이(운영자)가 교수에게 직접 컨택해 도입**한다. 즉:

- **수업·수강생 세팅은 레이가 전부 해서 드린다** → 교수는 `claimCourse`·학생 등록 같은 셋업을 **하지 않는다**.
- 교수는 로그인하면 **자기 수업이 바로 보이고**, **"보는 것"이 중심**이다. 작성(공지·설문)은 **최대한 간단**해야 한다.
- 핵심 질문: *"교수가 이 서비스를 쓰면서 보고 싶은 게 뭘까?"* → 참여율, 팀 구성, **팀 못 구한 학생**, 설문 결과.

따라서 교수 페이지는 **부담 최소화 + 한눈에 보기**가 원칙이다.

## 구현한 기능 (2026-06-14)

`tsc` 통과 · 테스트 50건 통과.

### A. 수업 현황 대시보드 (새 첫 탭 "현황")
로그인·수업 선택 시 가장 먼저 보이는 요약. 서버 `professor.dashboard`(집계 1콜).
- 수강생 수 / 프로필 완성(=실제로 시작한 학생) 수
- 진행 중 팀 / 완료 팀 수
- 팀 합류 학생 수 (전체 중)
- **팀 미배정 학생 수** (강조 카드 — 0보다 크면 amber)

### B. 팀 미배정 학생 목록
대시보드 안에서 **누가 아직 팀이 없는지** 실명·학과·학년으로 나열(+프로필 미완성 표시). 교수가 "너 팀 구했니?" 독려하거나 레이에게 알려 개입할 수 있게.

### C. 설문 응답률
대시보드에 설문별 **응답자/수강생 (비율%)** 막대. 독려 시점 판단용. (마감 설문은 (마감) 표기.)

### D. 설문 템플릿 원클릭
설문 빌더 부담 제거. 버튼 하나로 제목+문항 자동 채움(불러온 뒤 수정 가능):
- **중간 강의 만족도** (척도 3 + 주관식 1)
- **팀플 운영 점검** (척도 2 + 객관식 1 + 주관식 1)

### E. 클레임 UI 정리 (부담 최소화)
교수는 보통 세팅이 끝나 있으므로, **담당 수업이 있으면 "수업 검색" 입력을 기본 접어 둔다**. "+ 다른 수업 담당 등록"을 눌렀을 때만 펼침. (수업 없을 때만 검색이 바로 보임.)

## 교수가 보고 싶은 것 — 분석

| 관점 | 지금 보여주는 것 |
|---|---|
| "얼마나 참여하나" | 수강생 수 · 프로필 완성률 · 설문 응답률 |
| "팀은 잘 짜였나" | 진행/완료 팀 수 · 팀 합류 인원 · **팀 미배정 학생** |
| "학생 피드백" | 설문 결과(평균·분포·주관식, 기존) + 응답률(신규) |
| "공지 전달" | 공지 작성(원클릭) → 학생 CourseDetail 노출(기존) |

## 산출물 제출 — 학생 결과물 → 교수 한눈에 (2026-06-14 추가)

**요구**: "세팅하면 학생들이 중간 결과물을 올리고 교수가 한눈에 파악."

**구조** (파일 업로드 대신 외부 링크로 인프라 단순화):
- 교수가 **제출 항목(마일스톤)** 생성 — 제목·설명·마감(선택). 예: "1차 기획안".
- 각 팀이 **링크(구글드라이브·노션·깃허브) + 메모**로 제출 (마일스톤당 팀당 1개, 재제출=수정).
- 교수 **"제출" 탭**: 마일스톤별 **팀 제출 현황 매트릭스** — 제출/미제출 한눈에, 링크 열기, **"확인함" 체크**(재제출 시 자동 리셋).

**구현**:
- 스키마 0010: `course_milestones`, `team_submissions`(unique `milestoneId`+`teamId`)
- 서버 db: `createMilestone`·`getCourseMilestones`·`deleteMilestone`·`getCourseSubmissions`·`getTeamMilestones`·`submitDeliverable`(onDuplicateKeyUpdate)·`setSubmissionReviewed` + 라우터(`professor.*` 5 + `deliverables` 2)
- 클라: 교수 "제출" 탭(생성+매트릭스), 학생 TeamDetail "제출 항목" 섹션(링크+메모 제출)

**교수 시나리오**: "제출" 탭 → "1차 기획안: 4/6팀 제출. C·E팀 미제출. A팀 링크 열어보고 확인 체크" → 산출물 수합이 한 화면.

## 추후 후보 (효용은 있으나 비용/우선순위로 보류)

- **팀별 진척도**: 각 팀 일정 완료율·마감 임박. 데이터는 있음(`team_events`). 교수가 거기까지 볼지 미지수 → 수요 확인 후.
- **결과 CSV 내보내기**: 설문·수강생 명단 다운로드. 파일 다운로드 권한·포맷 정리 필요.
- **공지 푸시/알림**: 현재는 앱 내 노출만. 이메일·알림 연동은 별도 인프라.
- **학생 개별 활동 로그**: 과도한 추적 — 프라이버시·효용 균형상 보류.

이 항목들은 **교수에게 실제 도입해 본 뒤** 수요를 보고 결정하는 게 맞다(추측 구현 회피).
